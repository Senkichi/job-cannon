### 🚧 v5.0 Public Release Foundation — Cascade Audit + Strangerify P1 + PyPI (Phases 35-45)

**Started:** 2026-05-13
**Granularity:** standard (11 phases derived from natural plan-chunk boundaries)
**Coverage:** 60/60 forward requirements mapped (5 retroactively-validated requirements excluded — execution already shipped)

**Source plans:**
- `.planning/plans/2026-05-13-local-cascade-audit-plan.md` (cascade audit; 5 chunks → Phases 35-37 + canary collapsed into 40)
- `.planning/public-release/PLAN-P1.md` (Strangerify; 6 chunks → Phases 38-43)
- `.planning/public-release/DESIGN.md` (full P1–P4 design; P2 absorbed → Phases 44-45)

**Sequencing decisions documented:**

- **Canary timing (constraint 4 of milestone brief):** The audit's 1-week production canary (AUDIT-15/16) and config rewire (AUDIT-13/14) are **collapsed into Phase 40 (Workload Tiers + Cascade Rewire + Canary)** rather than executed standalone after Phase 37. Rationale: (1) Audit Chunks 1-4 produce the Case A/B decision (AUDIT-12) without touching `config.yaml`; (2) Strangerify Chunk 3 must rewrite the `providers:` block anyway (rename `low`/`mid`/`high` → `quick`/`score`/`triage`); (3) a single sequenced `config.yaml` Edit-tool session avoids two rewrites of a file that has been wiped three times by full-file writes; (4) canary observes the FINAL workload-class schema, which is what stays in production. Trade-off: canary window starts 4-5 phases into the milestone instead of mid-milestone. The audit *decision* is finalized in Phase 37; only wire-up + production observation slide to Phase 40.
- **Strangerify before PyPI:** PyPI publication (Phases 44-45) waits until Strangerify (Phases 38-43) is complete. Publishing a `pipx install job-cannon` that ships a broken stranger experience would burn the launch.
- **Foundation + Provider abstraction before workload tiers:** Phases 38-39 (FOUND-* + PROV-*) ship before Phase 40's tier rewrite. Foundation does not touch the tier system. Provider abstraction adds the new providers but keeps `low`/`mid`/`high` working until Phase 40 cuts over.
- **PyPI release pipeline could ship early in principle** (PYPI-01..03 are technically independent), but is placed in Phase 44 so the first published artifact carries post-Strangerify code, not pre-Strangerify code.

### Phase 36: Cascade Audit Eval Harness
**Goal**: An offline shadow-replay harness can compare LLM outputs across providers using a DeepSeek-V3.2 judge for any of the 6 non-scoring callsites.
**Depends on**: Phase 35 (audit harness reads cost telemetry to identify high-volume callsites)
**Requirements**: AUDIT-05, AUDIT-06, AUDIT-07, AUDIT-08
**Success Criteria** (what must be TRUE):
  1. With `OPENROUTER_API_KEY` set, the judge CLI returns a pairwise verdict with position-swap applied (`(A,B)` and `(B,A)` both judged; agreement counts as a confident verdict)
  2. `evals/cascade_audit/corpus_loader.py` replays N production DB rows for any of the 6 callsites with identical inputs the production code saw
  3. Per-callsite adapters exist for all 6 non-scoring callsites: `parse_structured_fields`, `find_careers_url`, `extract_jobs`, `description_reformat`, `company_research`, `ai_nav_discovery` — verifiable by an enumeration command listing all six
  4. Harness produces JSON verdict ADTs (winner/loser/tie with rationale) consumable by the report generator
  5. Judge calls hit OpenRouter's `:free` DeepSeek-V3.2 endpoint; no Anthropic spend incurred during eval
