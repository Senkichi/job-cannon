# v1.1 Design Spec: Harden & Resume Intelligence

## Overview

v1.1 hardens the shipped v1.0 MVP — fixing bugs, closing tech debt, polishing UX — and makes a deep investment in resume generation quality by letting the user seed the system with their existing resumes for both content and style guidance. A lightweight instrumentation layer collects user behavior data to inform v1.2 workflow optimization.

**Milestone name:** Harden & Resume Intelligence
**Scope:** 4 workstreams, ~20 discrete changes
**New dependency:** PyMuPDF (fitz) for PDF text extraction

## Workstream 1: Tech Debt & Bug Fixes

### 1.1 ATS Scanner Error Handling

**Problem:** `ats_scanner.py` caches HTTP errors (timeouts, 5xx) as permanent "miss" status. A temporary API outage looks like "company doesn't use this ATS" forever.

**Design:**
- Add `probe_status` distinction: `"miss"` (404, confirmed not found) vs `"error"` (timeout, 5xx, network failure).
- On `"error"`: set a `retry_after` timestamp with exponential backoff (1hr → 4hr → 24hr, max 3 retries).
- After 3 consecutive errors, downgrade to `"miss"` with `miss_reason = "unreachable"` so the user can see why in the companies UI.
- Existing `"miss"` from 404 responses stays permanent (company genuinely doesn't use that ATS).

**Files:** `job_finder/web/ats_scanner.py`, migration in `job_finder/web/db_migrate.py`

### 1.2 Indeed Email Parser

**Problem:** `job_finder/parsers/indeed_parser.py` is a non-functional stub that silently returns empty results. Indeed job alert emails are dropped with no user warning.

**Design:**
- Implement real parser following LinkedIn/Glassdoor pattern: BeautifulSoup HTML extraction with fallback strategies.
- Requires 2-3 sample Indeed alert emails to characterize the HTML format.
- Wire into `sources.gmail.senders.indeed` config key (already exists, currently empty string).
- If parser fails to extract jobs from an email, log a warning and surface it in the dashboard activity feed ("1 Indeed email could not be parsed") instead of silent drop.

**Files:** `job_finder/parsers/indeed_parser.py`, `job_finder/web/pipeline_runner.py` (ingestion path)

### 1.3 Remove Deprecated jd_fetcher.py

**Problem:** `job_finder/web/jd_fetcher.py` was replaced by cost-ordered enrichment in Phase 10 but left in the codebase. Risk of maintainer confusion.

**Design:** Delete the file. Remove any remaining imports — notably `tests/test_scoring.py` which imports it in 7 locations and is the only remaining consumer. `data_enricher.py` fully replaces it. The test functions that test JD fetching behavior should be deleted since `data_enricher.py` has its own test coverage; if any edge cases are unique to the old tests, port them to test `data_enricher` instead.

**Files:** `job_finder/web/jd_fetcher.py` (delete), `tests/test_scoring.py` (delete jd_fetcher tests or port to data_enricher)

### 1.4 Dedup Normalizer SQL Hygiene

**Problem:** `dedup_normalizer.py` uses f-string SQL interpolation for table/column names in `_update_fk_references()` (line 551). The table/column names come from a hardcoded Python list (lines 540-546), NOT from user input, and all data values are properly parameterized — so there is no actual injection surface. However, the pattern is a code hygiene concern: future edits could inadvertently introduce user-controlled values into the f-string.

**Design:** Replace f-string table/column interpolation with an explicit allowlist assertion. Add a comment documenting why the pattern is safe. Low-effort change to prevent future mistakes.

**Files:** `job_finder/web/dedup_normalizer.py`

### 1.5 Repeated Per-Job Queries in Expand Route

**Problem:** The job expand route (`jobs.py:expand()`) runs separate queries for `resume_generations` and `interview_preps` per job expansion. Currently this is 1+1+1 per user click (not a true N+1 since expand is one-at-a-time), but the same pattern is repeated in the re-score and quick-apply routes, tripling the boilerplate.

**Design:** Extract a `load_job_context(conn, dedup_key)` helper that batch-fetches the job row, resume history, and interview prep in a single function. Reduces code duplication across `expand()`, `rescore()`, and `quick_apply()` routes. Not a performance fix — a maintainability improvement.

**Files:** `job_finder/web/blueprints/jobs.py`

### 1.6 Pipeline Detector Edge-Case Tests

**Problem:** Missing test coverage for edge cases identified in CONCERNS.md.

**Design:** Add ~6 tests to `tests/test_pipeline_detector.py`:
- Null `company_name` in job record
- Empty title string
- Overlapping timing windows (rejection + confirmation emails within 2 hours of each other)
- Single-word company substring false positive (e.g., "Meta" appearing in "metadata")
- ATS domain signal without `detection_type` classification
- Null `first_seen` date in job record

**Files:** `tests/test_pipeline_detector.py`

### 1.7 Persistent Log Files

**Problem:** All 144 log calls across 19 modules go to stderr only. Logs are lost when the terminal closes.

**Design:**
- Configure a `RotatingFileHandler` in `create_app()` writing to `logs/job_finder.log`.
- Rotate at 5MB, keep 3 backups (~15MB max disk usage).
- INFO level to file. Existing stderr behavior unchanged.
- Add `logs/` to `.gitignore`.

**Files:** `job_finder/web/__init__.py`, `.gitignore`

### 1.8 CONCERNS.md Cleanup

**Problem:** Three entries in `.planning/codebase/CONCERNS.md` describe issues that were already fixed or never existed:
- Phase 9 Resume OAuth Fix — shipped in Phase 9 (`gmail_auth.py` auto-upgrades tokens, `drive_status.py` provides diagnostics)
- Phase 8.3 Manual ATS Scan Pending Companies — shipped in Phase 8 (`companies.py:scan()` calls `probe_ats_slugs()` before `run_ats_scan()`)
- Dedup Normalizer Fuzzy String Matching — `thefuzz` is not imported or used; normalization is purely deterministic regex

**Design:** Remove the three stale entries. No code changes.

**Files:** `.planning/codebase/CONCERNS.md`

## Workstream 2: UX Fixes

### 2.1 Re-score Button Visual Feedback

**Problem:** Clicking re-score gives no indication that anything is happening.

**Design:**
- On click, immediately swap the button to a loading state: spinning indicator + "Scoring..." text.
- Use `hx-disabled-elt` or `hx-indicator` on the re-score button.
- On completion, the response fragment replaces the button area with the updated score + rationale.
- On error, show inline error message in place of the button.

**Files:** `job_finder/web/templates/jobs/_row_expanded.html`, `job_finder/web/blueprints/jobs.py` (re-score route response)

### 2.2 Re-score Updates Compact Row Score

**Problem:** After re-scoring, the expanded section updates but the compact row above still shows the old score.

**Design:**
- The re-score endpoint's response includes an `hx-swap-oob` element that targets the compact row's score `<td>` by ID.
- The compact row's score cell gets a unique ID based on the job's `dedup_key` (e.g., `id="score-{dedup_key}"`).
- Main swap updates the expanded section as before. Out-of-band swap updates the compact row score simultaneously.

**Files:** `job_finder/web/templates/jobs/_row_expanded.html`, `job_finder/web/templates/jobs/_row.html` (compact row), `job_finder/web/blueprints/jobs.py`

### 2.3 jd_full Display in Detail View

**Problem:** The expanded row shows the description snippet but not `jd_full` even when it exists.

**Design:**
- Add a collapsible "Full Job Description" section in the expanded row, below the description snippet.
- Only renders when `jd_full` is present and non-empty.
- Collapsed by default. Click header to expand/collapse (pure HTML `<details>/<summary>` element — no JS needed).
- Content displayed with `white-space: pre-wrap` for whitespace preservation.

**Files:** `job_finder/web/templates/jobs/_row_expanded.html`

## Workstream 3: Resume Template & Style Guidance

### 3.1 PDF Upload & Extraction

**Problem:** The resume generation system has no way to learn from the user's existing resumes. It generates from scratch using only `experience_profile.json` and Claude's general resume writing knowledge.

**Design:**

**New dependency:** `PyMuPDF` (fitz) for PDF text extraction. Lightweight, no Java dependency, handles Google Docs PDF exports well.

**New route:** `POST /profile/resume-upload`
- Accepts one or more PDF files via multipart form upload.
- Extracts text from each PDF using PyMuPDF.
- Sends extracted text to Claude Opus for extraction. Reuses the existing `extract_profile_from_markdown()` pattern in `profile_schema.py` (which already does Opus-powered profile extraction) as the foundation, extended with style extraction. Two extraction goals:
  1. **Content extraction:** Positions, achievements, skills, education. Compared against current `experience_profile.json`.
  2. **Style extraction:** Section ordering, summary tone, bullet point style, date format, skills grouping, phrasing patterns.
- Stores raw PDF text extractions in `data/resume_uploads/{filename}_{timestamp}.txt` for reference (never injected into generation prompts). The `data/resume_uploads/` directory is created on first upload and added to `.gitignore`.

**Content synthesis (conflict resolution):**
When extracted content differs from `experience_profile.json` (e.g., a bullet refined in the PDF vs an older version in the profile), the system does NOT keep both and does NOT blindly prefer one source. Instead:
- Claude produces a single synthesized version per conflicting item, combining the strongest qualities of both (metrics from one, framing from the other, best verb choice).
- The user sees a review screen per conflict:
  - Profile version (left)
  - PDF version (right)
  - Proposed synthesis (below, highlighted)
  - Accept / Edit / Skip per item
- On accept, the synthesis replaces both in `experience_profile.json`. The profile stays singular — one canonical version per item, no accumulating variants.
- Raw extractions are archived in `data/resume_uploads/` but never enter the prompt path.
- Subsequent PDF uploads synthesize against the current canonical profile, not against previous uploads. Always "current profile vs new input."

**Files:** New `job_finder/web/resume_uploader.py`, `job_finder/web/blueprints/profile.py` (new routes), new template partials

### 3.2 Resume Style Guide

**Problem:** No mechanism to capture and persist the user's resume style preferences in a structured format.

**Design:**

**New file:** `resume_style_guide.json` in the project root (alongside `experience_profile.json`). Added to `.gitignore`.

**Schema:**
```json
{
  "summary_style": "concise, third-person, metrics-forward",
  "bullet_style": "action-verb led, quantified where possible, 1-2 lines max",
  "section_order": ["summary", "skills", "experience", "education"],
  "date_format": "Mon YYYY - Mon YYYY",
  "skills_grouping": "categorized by domain",
  "tone": "confident but not boastful, emphasizes impact over activity",
  "phrasing_examples": [
    {"pattern": "Led X resulting in Y", "context": "leadership bullets"},
    {"pattern": "Built/Designed X serving Y users", "context": "technical bullets"}
  ],
  "extracted_from": ["resume_2024_stripe.pdf", "resume_2025_general.pdf"],
  "last_updated": "2026-03-12"
}
```

This is descriptive, not prescriptive — Claude interprets these as style guidance, not rigid formatting rules.

**Multi-PDF merge:** When multiple PDFs are uploaded, style signals are merged. Consistent patterns across resumes become strong signals (e.g., if 3/3 use action-verb bullets, that's definitive). Conflicting style signals are noted with the most recent PDF's style taking slight precedence.

**Files:** New `resume_style_guide.json`, `job_finder/web/resume_uploader.py` (writes this file)

### 3.3 Wire Style Guide Into Generation Prompts

**Problem:** Two existing mechanisms are broken or disconnected:
1. `resume_preferences.summary_style` and `resume_preferences.emphasis` fields exist in the profile schema but are never injected into Claude prompts.
2. No way to inject persistent style guidance into resume generation.

**Design:**

**Prompt injection priority order** (highest priority wins on conflict):
1. JD requirements (from `jd_full`)
2. Fit analysis (from Sonnet evaluation — `resume_priority_skills`, `strengths`)
3. Resume style guide (from `resume_style_guide.json`)
4. Profile resume preferences (`summary_style`, `emphasis` from `experience_profile.json`)
5. Drive feedback preferences (from `resume_preferences_detected` table)

**Conflict resolution in the prompt:** Priority is enforced via prompt structure, not omission. All five sources are included when available. Higher-priority sources appear earlier in the prompt with explicit precedence instructions: "When style guide and JD requirements conflict, always follow JD requirements." Claude naturally weights earlier, more prominent instructions higher. Lower-priority sources serve as fallback guidance for aspects not addressed by higher-priority sources.

**Implementation:**
- Load `resume_style_guide.json` in `resume_generator.py` alongside the experience profile.
- Inject as a "Style Reference" section in the system prompt, after the closed-world constraint but before the user message, with a precedence note: "These style preferences are soft guidelines. JD requirements and fit analysis take priority."
- Wire up `resume_preferences.summary_style` and `resume_preferences.emphasis` from `experience_profile.json` into the user message (currently dead fields).
- The Drive feedback loop (`resume_feedback.py`) continues working unchanged — detected preferences from edits layer on top of the base style guide at priority level 5.

**Files:** `job_finder/web/resume_generator.py` (prompt construction), `job_finder/web/profile_schema.py` (style guide loader)

### 3.4 Resume Style UI

**Problem:** No interface for uploading PDFs or reviewing/editing the extracted style guide.

**Design:**

New section on the `/profile` page: "Resume Style"

**Components:**

**Upload area:**
- Drag-and-drop zone or file picker for PDF files on the `/profile` page.
- `POST /profile/resume-upload` accepts multipart form data.
- Shows spinner during extraction (Opus call may take 10-20s).
- On completion, redirects to the content review screen.

**Content review screen** (`GET /profile/resume-review/<upload_id>`):
- A new page (not an HTMX partial) showing all extracted items that differ from the current profile.
- Each conflict is a card with three sections:
  - "Current Profile" (left column, gray background)
  - "From PDF" (right column, gray background)
  - "Proposed Synthesis" (full width below, highlighted background)
- Each card has three buttons: Accept (saves synthesis to profile), Edit (opens inline `<textarea>` pre-filled with the synthesis — user modifies and clicks Save), Skip (leaves profile unchanged for this item).
- Extraction results are stored temporarily in a `resume_upload_reviews` table with `status = 'pending'`. Each item row has `profile_version`, `pdf_version`, `synthesis`, and `accepted` columns.
- Actions are individual `POST /profile/resume-review/<upload_id>/<item_id>` calls via HTMX — each card is self-contained, no batch submit needed.
- No progress persistence needed across browser sessions — if the user closes the browser, pending review items remain in the table and the review page can be reopened from `/profile`.
- When all items are accepted/skipped, the upload is marked complete and the user returns to `/profile`.

**Style guide display:**
- Read-only summary on `/profile` page showing current style guide fields in human-readable form.
- Shows which PDFs contributed ("Extracted from: resume_2024.pdf, resume_2025.pdf").
- "Edit Style Guide" link opens an edit view.

**Style guide edit** (`GET /profile/style-guide/edit`):
- Simple form with one field per style guide key (text inputs for short fields, textareas for `phrasing_examples`).
- `POST /profile/style-guide` saves to `resume_style_guide.json`.
- HTMX form submission, returns to the display view on save.

**Files:** `job_finder/web/templates/profile/` (new partials: `_resume_upload.html`, `_resume_review.html`, `_style_guide.html`, `_style_guide_edit.html`), `job_finder/web/blueprints/profile.py` (new routes), `job_finder/web/db_migrate.py` (migration for `resume_upload_reviews` table)

## Workstream 4: Workflow Instrumentation

### 4.1 Activity Tracking Table

**Problem:** No structured data on how the user interacts with the app. System logs capture what the app does; nothing captures what the user does.

**Design:**

**New table:** `user_activity` (migration 9)
```sql
CREATE TABLE user_activity (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    action TEXT NOT NULL,
    target_id TEXT,
    timestamp TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_user_activity_action ON user_activity(action);
CREATE INDEX idx_user_activity_timestamp ON user_activity(timestamp);
```

**Tracked actions:**
| Action | Target | Trigger |
|--------|--------|---------|
| `sync` | null | User clicks "Sync Now" |
| `expand_job` | dedup_key | User expands a job row |
| `status_change` | dedup_key | User changes pipeline status |
| `paste_jd` | dedup_key | User pastes a job description |
| `rescore` | dedup_key | User triggers re-score |
| `generate_resume` | dedup_key | User generates a resume |
| `quick_apply` | dedup_key | User uses Quick Apply |
| `batch_score` | "haiku" or "sonnet" | User starts batch scoring |

**Helper function:**
```python
def log_activity(action: str, target_id: str | None = None) -> None:
    """Log a user action. Uses Flask g.db when in request context."""
    from job_finder.web.db_helpers import get_db
    from flask import current_app
    conn = get_db(current_app.config["DB_PATH"])
    conn.execute(
        "INSERT INTO user_activity (action, target_id) VALUES (?, ?)",
        (action, target_id),
    )
```

Uses Flask request-scoped `g.db` connection (same pattern as all route handlers). One-line call added to each relevant route handler. Append-only, no reads during normal operation. For the `sync` action which may delegate to a background thread, the `log_activity` call happens in the route handler before the thread spawns, not inside the thread.

**No UI in v1.1.** Data collection only. v1.2 will add a "Usage Insights" section to the dashboard.

**Files:** `job_finder/web/db_migrate.py` (migration 9), new `job_finder/web/activity_tracker.py`, 6-8 blueprint files (one-line insertions)

CREATE TABLE resume_upload_reviews (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    item_type TEXT NOT NULL,        -- 'position', 'achievement', 'skill', 'education'

## Success Criteria

1. **Tech debt:** ATS scanner retries on errors and distinguishes from real misses. Indeed parser extracts jobs from Indeed alert emails. No stale code or docs remain.
2. **UX:** Re-score shows loading state, updates both expanded and compact row. Full JD viewable in expanded row.
3. **Resume:** User can upload existing resume PDFs. Content synthesized into canonical profile with review. Style guide extracted and injected into all future resume generations.
4. **Instrumentation:** Key user actions logged to `user_activity` table. System logs persisted to rotating file.
