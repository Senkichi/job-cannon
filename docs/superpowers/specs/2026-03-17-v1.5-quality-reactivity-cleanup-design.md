## Phase 1: Data Quality

### JD Quality Audit & Fix

**Problem:** Job descriptions are visibly broken across multiple failure modes. Known examples: Senior Data Scientist at Microsoft, Data Scientist Lead at Kontakt.io, Senior Data Scientist at Vanta.

**Approach:**
1. Query DB for raw `description` and `jd_full` fields of the 3 named examples to identify specific failure mode
2. Write diagnostic queries to detect common failure signatures across all jobs:
   - Empty `jd_full` where enrichment should have populated it
   - HTML entities in `description` (e.g., `&amp;`, `&lt;`)
   - Truncated content (description < 50 chars for jobs that should have full content)
   - Encoding issues (garbled unicode, mojibake)
3. Categorize failures by root cause: parser bug, fetcher bug, rendering bug, data corruption
4. Fix each root cause in the appropriate layer (parsers/, sources/, templates/, filters)
5. One-time cleanup migration for existing broken data
6. Add guards to prevent recurrence (validation on ingest, logging on suspicious content)

**Files:** `job_finder/parsers/`, `job_finder/sources/`, `job_finder/web/templates/jobs/`, `job_finder/web/blueprints/jobs.py`

### JSON Deserialization Hardening

**Problem:** Multiple modules deserialize JSON from SQLite TEXT columns without consistent error handling. `db.py` lines 107-108 (`sources` and `source_urls` fields) lack try/except.

**Approach:**
1. Add `safe_json_load(raw: str | None, default: Any = None) -> Any` utility to `job_finder/web/db_helpers.py`
2. Apply to unprotected call sites in `db.py` (lines 107-108)
3. Verify existing protected sites (haiku_scorer.py, db.py locations_raw) follow consistent pattern
4. Search for any other unprotected `json.loads()` calls across the codebase

**Files:** `job_finder/web/db_helpers.py`, `job_finder/db.py`

### Success Criteria
- [ ] All 3 named example jobs display correctly
- [ ] Diagnostic queries find no remaining broken descriptions in DB
- [ ] All `json.loads()` calls use `safe_json_load()` or have explicit try/except
- [ ] Tests pass

## Phase 2: HTMX Reactivity

**Problem:** Many HTMX actions don't update the DOM to reflect changes without a full page refresh. The app should feel reactive — every user action immediately reflected in UI.

### Score Cell OOB Update Verification

Verify the existing hidden-div OOB pattern (`_row_expanded.html:217-223`) actually updates the compact row's score cell correctly (number + color class). If it works, no changes needed. If other compact row elements need updating, add additional OOB swap targets.

### Row Reordering on Score Change

**Approach:** HX-Trigger table re-fetch.
1. After rescore or paste-jd, include `HX-Trigger: scoreUpdated` response header from the route
2. Add `hx-trigger="scoreUpdated from:body"` to the filter form in `index.html` (or a hidden element)
3. This triggers a re-fetch of `/jobs/table` with current sort/filter params
4. Result: table body re-renders with correct sort order, reflecting new score

**Trade-off:** Re-fetches entire table body. Acceptable for typical table sizes (<200 rows). Avoids fragile client-side DOM manipulation of compact+expanded row pairs.

### Full HTMX Audit

Walk through every fragment route and verify the response updates all visually affected elements:
- Status dropdown changes
- Dismiss/return actions
- Settings saves
- Interview prep generation status
- Any other interactive elements

For each: verify `hx-target`, `hx-swap`, and response content are correct. Fix any that require page refresh.

### Success Criteria
- [ ] Re-scoring a job updates score in compact row without page refresh
- [ ] Re-scoring a job re-sorts the table when sorted by score
- [ ] Status changes reflect immediately in both compact and expanded views
- [ ] No HTMX action requires a manual page refresh to see results

## Phase 3: UI Features + Button Feedback

### Save JD Without Scoring

**Problem:** Only way to save a pasted JD is via "Score with JD" which triggers AI scoring. Users may want to persist JD for later review without cost.

**Approach:**
1. Add "Save" button next to "Score with JD" in `_row_expanded.html` paste form
2. New route `POST /jobs/<key>/save-jd` in `jobs.py`:
   - Updates `jd_full` column in DB
   - Returns updated expanded row (JD collapsible section now shows content)
   - No AI calls, no budget impact
3. "Save" button gets spinner + `hx-disable-elt="this"` for feedback

**Files:** `job_finder/web/templates/jobs/_row_expanded.html`, `job_finder/web/blueprints/jobs.py`

### Universal Button Feedback

**Problem:** Not all buttons have visual feedback (spinner, disable) during their action.

**Approach:**
Audit every interactive button across all templates. Each must have:
- `hx-disable-elt="this"` to prevent double-clicks
- `.htmx-indicator` spinner showing action in progress
- Response that visually confirms the action completed (no page refresh)

**Buttons to audit:**
- Rescore (already has feedback)
- Score with JD (already has feedback)
- Save JD (new — add feedback)
- Dismiss/return buttons
- ATS scan trigger
- Settings save
- Interview prep generation
- Any other interactive buttons

### Smooth Scroll on Expand/Collapse

**Approach:**
1. Add global `htmx:afterSwap` event listener in `index.html`
2. After expand/collapse swap within the job table, call `element.scrollIntoView({ behavior: 'smooth', block: 'nearest' })`
3. Use `block: 'nearest'` to avoid jarring scroll when element is already visible
4. Scope to job table swaps only (check event target is within `#job-table-body`)

**Files:** `job_finder/web/templates/jobs/index.html`

### Success Criteria
- [ ] "Save" button persists JD without triggering scoring
- [ ] Every button shows spinner/disable during its action
- [ ] Every button action returns visible confirmation without page refresh
- [ ] Expanding a row below the fold scrolls it into view smoothly
