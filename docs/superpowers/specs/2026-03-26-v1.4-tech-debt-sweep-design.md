### 20.3: Models.py layer fix

**Problem:** `job_finder/models.py` (foundation layer) imports `normalized_dedup_key` from `job_finder.web.dedup_normalizer` (web layer), inverting the dependency direction.

**Fix:** Move `normalized_dedup_key()` into `models.py` as `Job.normalized_dedup_key()` static method. Update all importers (~8 call sites).

### 20.4: SQLite helper extraction

**Problem:** 35+ occurrences of `conn = sqlite3.connect(db_path); conn.row_factory = sqlite3.Row` across 15+ files.

**Fix:** Add `standalone_connection(db_path)` context manager to `db_helpers.py`:

```python
@contextmanager
def standalone_connection(db_path: str):
    """Context manager for background/CLI sqlite3 connections.
    Sets row_factory=Row and WAL mode. NOT for Flask request handlers (use g.db).
    """
    conn = sqlite3.connect(db_path)
    conn.row_factory = sqlite3.Row
    conn.execute("PRAGMA journal_mode=WAL")
    try:
        yield conn
    finally:
        conn.close()
```

Replace all 35+ occurrences across 15 files. See `.planning/todos/pending/2026-03-26-extract-sqlite-connect-helper.md` for the full file list, or grep for `sqlite3.connect` to find all sites.

### 20.5: Config thread-safety

**Location:** `job_finder/web/blueprints/settings.py:154`

**Problem:** `settings.py` replaces the shared `JF_CONFIG` dict while background threads may be reading it. Low risk in practice (single-user, GIL makes dict replacement atomic), but the pattern is technically unsafe.

**Fix:** Add `get_config_snapshot()` helper:

```python
import copy

def get_config_snapshot(app) -> dict:
    """Return a frozen deep copy of JF_CONFIG for use in background threads."""
    return copy.deepcopy(app.config.get("JF_CONFIG", {}))
```

Background threads (APScheduler jobs) call `get_config_snapshot()` at job-start instead of reading individual keys from the shared dict across multiple statements.

### Split constraints

- New blueprint modules register on the same URL prefix as the parent (e.g., batch_scoring and sync register under `/dashboard/`)
- New blueprint `name` values: `batch_scoring`, `sync`, `resume_review`, `guidelines`. All `url_for()` references in templates and Python code must be updated to use the new blueprint names (e.g., `url_for('batch_scoring.start')` instead of `url_for('dashboard.batch_score_haiku_start')`)
- Non-blueprint extractions (enrichment_tiers.py, scoring_runner.py, etc.) are plain Python modules
- Re-exports for backwards compatibility must be removed within Phase 22 itself — all importers are updated in the same phase, so no transition period is needed
- Every test file updates its imports to point at the new module locations

## Phase 23: N+1 Query Batching

Five patterns batched with per-job error handling (log failures, continue processing):

### 23.1: Haiku scoring batch read

**File:** `scoring_runner.py` (post-Phase-22 location; pre-split: `pipeline_runner.py`)

**Current:** Individual `SELECT` per job in Haiku scoring loop.
**Batched:** Single `SELECT ... WHERE dedup_key IN (?)` for the batch. Iterate results in-memory.

### 23.2: Sonnet evaluation batch read

**File:** `scoring_runner.py` (post-Phase-22 location; pre-split: `pipeline_runner.py`)

Same pattern as 23.1.

### 23.3: Stale detector batch update

**File:** `stale_detector.py`

**Current:** Per-row `update_pipeline_status()` call (4N operations per archive batch).
**Batched:** Single `UPDATE jobs SET pipeline_status = 'stale' WHERE dedup_key IN (?)` + batch event insert. All-or-nothing per batch (all jobs get the same status).

### 23.4: Dashboard cancellation check

**File:** `batch_scoring.py` (post-Phase-22 location; pre-split: `dashboard.py`)

**Current:** Per-job `SELECT cancelled FROM batch_sessions WHERE id = ?` inside the scoring loop.
**Batched:** Check once before processing each job (or once per N jobs). The session's cancelled state doesn't change per-job — it's set by a separate cancel request.

### 23.5: Session counter deferred update

**File:** `batch_scoring.py` (post-Phase-22 location; pre-split: `dashboard.py`)

**Current:** Per-job session counter `UPDATE + COMMIT`.
**Batched:** Increment counter once after the full batch completes, not per-job. Intermediate progress is already tracked by the polling endpoint via a separate mechanism.
