# v1.3 Quality & Calibration Milestone Design

## Overview

Full quality pass across four workstreams: test infrastructure, parsers/scrapers, company data pipeline, and scoring calibration. Implements all C1-C4 recommendations from the Phase 23 scoring evaluation.

**Milestone:** v1.3 Quality & Calibration
**Phases:** 24-27
**Order:** Tests -> Parsers -> Companies -> Scoring

---

## Phase 24: Test Audit & Restructure

### Goal

Ensure every test is properly isolated (no real file/config/DB side effects), remove redundant tests, fill coverage gaps, and restructure test files for clarity.

### Workstreams

**1. Side-effect audit**

Scan all 31 test files for:
- Writes to real filesystem paths without `tmp_path`/`monkeypatch` (the profile wipe class of bug where `test_post_profile_save_redirects_on_success` was overwriting real `experience_profile.json` with Acme placeholder data)
- Tests that import and call production code touching `config.yaml`, `experience_profile.json`, `resume_style_guide.json`, or the live `jobs.db` without path isolation
- Tests that make real HTTP/API calls without mocking (Gmail API, SerpAPI, Anthropic, DuckDuckGo)

**2. Redundancy audit**

Identify tests that:
- Test the same behavior from multiple angles without added value
- Were written for intermediate states of the code that no longer exist
- Test internal implementation details rather than behavior

**3. Coverage gaps**

Check for untested:
- Error paths in production code
- Edge cases in parsers (empty bodies, malformed HTML)
- Integration points between modules (pipeline_runner -> haiku_scorer -> sonnet_evaluator flow)

**4. Restructure**

Standardize:
- Fixture organization in `conftest.py` (currently 7 fixtures; some test files define their own that should be shared)
- Split oversized test files if any exceed ~500 tests
- Naming conventions and test class organization

### Success Criteria

- `pytest tests/` passes with zero side effects on real files
- No test creates, modifies, or deletes files outside `tmp_path`/temp directories
- All tests identified in the redundancy audit are addressed (removed, or documented as intentional)
- Coverage gaps documented and filled

## Phase 25: Parser & Scraper Audit

### Goal

Verify all parsers produce correct output against current email formats, fix any parsing failures, and ensure error handling is robust.

### Workstreams

**1. Parser correctness audit**

**Prerequisite:** User provides recent email samples from each active alert source. If samples are unavailable for a source, that parser is audited code-only.

For each of the 4 email parsers (LinkedIn, Glassdoor, Indeed, ZipRecruiter):
- Test against user-provided email samples to verify they extract correct title, company, location, salary
- Compare output against test fixtures to catch format drift
- Flag any parser that fails on current real emails

**2. Parser robustness review**

- **Glassdoor** (`glassdoor_parser.py`, 173 lines): CSS class name fragility (`gd-628b46d9ce`, `gd-6c2846d4dc`) — verify these still match current email format
- **ZipRecruiter** (`ziprecruiter_parser.py`, 356 lines): Generic HTML heuristics — effectiveness without sample emails, placeholder filter robustness
- **Indeed** (`indeed_parser.py`, 546 lines): Dual-strategy (text primary `engage.indeed.com` delimiters, HTML fallback) — when does fallback trigger? Edge cases in salary parsing (hourly -> annual conversion at 2080 hours)
- **LinkedIn** (`linkedin_parser.py`, 171 lines): Plain text parsing with separator lines — edge cases with unusual formatting, salary extraction ($XXK-$XXK)

**3. Source module audit**

- `gmail_source.py` (232 lines): Sender-to-parser routing complete? Parse failure archival to `data/parse_failures/` working?
- `serpapi_source.py` (125 lines): Salary extraction from `detected_extensions.salary`, description assembly from highlights — still match current SerpAPI response format?

**4. ATS scanner & careers scraper**

- `ats_scanner.py` (1,387 lines): Lever/Greenhouse/Ashby URL patterns still correct? Retry state machine (exponential backoff, 3 failures -> permanent unreachable) working correctly?
- `careers_scraper.py` (205 lines): ATS redirect detection, keyword filtering via `_title_matches()`

### Success Criteria

- All parsers correctly extract jobs from current real email samples
- No silent data loss (fields parsed as empty when data exists)
- Format drift issues identified and fixed
- Parse failure archival verified functional

## Phase 26: Company Backfill Review & Fix

### Goal

Verify the Phase 22 company backfill produced correct results on live data, fix any issues found, and ensure the full company pipeline (linking -> ATS probing -> enrichment) works optimally.

### Workstreams

**1. Live data verification** (read-only queries first)

- Count companies, linked jobs, unlinked jobs
- Spot-check fuzzy matches for false positives (unrelated companies merged)
- Check for duplicate company records that fuzzy matching should have caught
- Verify denylist: no company records for "Unknown", "Medical jobs", "Clinical jobs", "RemoteHunter", "Jobgether", "Mercor", "Crossing Hurdles"

**2. DDG enrichment quality check**

- Count companies with `homepage_url` populated vs empty
- Spot-check homepage URLs for correctness
- Industry/size data: populated? Reasonable values?
- Identify companies with garbage or wrong enrichment data

**3. Code quality audit**

- `fuzzy_match_company()`: threshold=85 tuning, short-name guard (< 4 chars) — "IBM", "SAP" handling
- `link_jobs_to_companies()`: race condition potential on concurrent runs, dedup during single run
- `run_ddg_enrichment()`: error handling, column safety (UPDATE only valid columns)
- `normalize_company()` in `dedup_normalizer.py`: suffix coverage (Inc., LLC, Corp., Ltd., etc.)

**4. Fix and re-run**

- Fix any code or data issues found
- Re-run backfill to clean up data quality issues
- Verify final state matches success criteria

### Success Criteria

- Zero unlinked jobs with a non-null, non-denylist company name (edge cases with empty/unparseable company names are documented as exceptions)
- No duplicate company records for the same real company
- Homepage URLs populated and correct where DDG returned results
- Fuzzy matching produces no false positives (unrelated companies merged)

## Phase 27: Scoring Calibration (C1-C4)

### Goal

Implement all four critical recommendations from the Phase 23 scoring evaluation to improve scoring accuracy.

### C1: Haiku Snippet Expansion (500 -> 2000 chars + skill keyword injection)

**Problem:** At 500 characters, Haiku sees only 6-17% of a job posting. Requirements/qualifications sections where skill-match signals live are truncated. Primary driver of false negatives.

**Changes:**
- `haiku_scorer.py`: Replace `description[:500]` (line 144) with `build_description_snippet()` function
- Snippet composition: first 1200 chars + requirements section extraction (regex for "requirements", "qualifications", "what you'll bring", etc.) + skill keyword match summary from full description
- Profile skills already available via `profile_section.get("skills", [])` in the existing code — no new parameter needed
- Max total: 2000 chars

**Cost impact:** +$0.03/month (400 additional tokens x 300 jobs x $0.25/MTok)

**Tests:**
- Snippet builds correctly for descriptions shorter than 1200 chars
- Requirements section extracted when present after char 800
- Skill keyword counts injected from full description
- No skill matches produces explicit "[No candidate skill keywords found]" note
- Max chars respected

### C2: Threshold Lowering + Borderline Re-evaluation Band

**Problem:** Ground truth job (Kikoff) scored Haiku 42 — filtered at threshold 55. Haiku is essentially random in the C-band (42-62); Sonnet scores for C-band jobs range 22-82.

**Changes:**
- `config.yaml`: `scoring.haiku_threshold` from 55 to 42
- `config.yaml`: New `scoring.borderline_band` block (enabled=true, low=42, high=54, reeval_chars=4000)
- `pipeline_runner.py`: Jobs scoring 42-54 get a **second Haiku call using the full job description** (up to `reeval_chars=4000`, vs the standard 2000-char snippet from C1). This gives Haiku maximum context to make a confident pass/filter decision before committing to the more expensive Sonnet call. The second call uses the same system prompt and schema as the initial call. If re-eval score < 42, the job is filtered. If >= 42, proceeds to Sonnet.
- The **re-eval score** replaces the original score in the database (it had more context, so it's more reliable). The re-eval score is what Sonnet threshold comparison uses.

**Cost impact:** +$0.22/month (10 additional Sonnet calls + 15 borderline re-evals)

**Tests:**
- Jobs below 42 filtered (no Sonnet call)
- Jobs 42-54 trigger borderline re-eval
- Borderline re-eval below 42 on second call: filtered
- Borderline re-eval >= 42 on second call: proceeds to Sonnet
- Jobs >= 55 proceed directly to Sonnet (no re-eval)
- Borderline band disabled: jobs >= 42 go straight to Sonnet

### C3: Pre-Haiku Exclusion Filter

**Problem:** `config.yaml` exclusions (title_keywords, companies) are defined but never read by scorers. Excluded-category jobs waste API budget.

**Changes:**
- New module `exclusion_filter.py` with `should_exclude(job, exclusions, min_salary)` function
- Title keyword matching (case-insensitive substring)
- Company name matching (case-insensitive exact)
- Optional salary floor check with 15% tolerance
- Called in `pipeline_runner.py` before Haiku scoring
- Returns `(bool, reason_str)` for logging

**Cost impact:** -$0.03 to -$0.05/month saved

**Tests:**
- Title keyword exclusion ("junior", "associate", etc.)
- Company exclusion
- Salary floor exclusion with tolerance
- No false exclusions on good jobs (e.g., "Senior Data Scientist" not excluded by "junior")
- Empty exclusions config handled gracefully

### C4: Sonnet Preference Injection

**Problem:** Sonnet receives only `experience_profile.json` (skills, positions) but never `config.yaml` preferences (target_titles, target_locations, min_salary). Evaluates "can do" but not "wants to do."

**Naming hazard:** `evaluate_job_sonnet()` has a parameter called `profile` (the experience profile dict from `experience_profile.json` — positions, skills, achievements) AND the config dict contains a `config["profile"]` key (the preferences section from `config.yaml` — target_titles, target_locations, min_salary). These are **different objects** with the same name. To prevent confusion, rename the function parameter from `profile` to `experience_profile` as part of this change.

**Changes:**
- Rename the `profile` parameter of `evaluate_job_sonnet()` to `experience_profile` for clarity. Update the single call site in `pipeline_runner.py` to match.
- Inside `evaluate_job_sonnet()`, extract preferences from `config.get("profile", {})` (target_titles, target_locations, min_salary, industries) and add a "Candidate Preferences" section to the user prompt after the experience profile
- Sonnet now evaluates both experience fit AND preference alignment

**Cost impact:** Negligible (small prompt addition)

**Tests:**
- Preferences section appears in Sonnet prompt when present in config
- Missing preferences handled gracefully (no crash, "Not specified")
- Sonnet still works with experience profile only (backward compat)

### Success Criteria

- All four changes implemented with tests
- Scoring pipeline: exclusion filter -> Haiku with expanded snippet -> borderline re-eval -> Sonnet with preferences
- Existing tests still pass (no regressions)
- Pipeline runner orchestrates the full flow correctly

## Cross-Phase Dependencies

- Phase 24 (Tests) has no dependencies and should run first — a clean test foundation ensures subsequent phases can rely on test results
- Phase 25 (Parsers) depends on Phase 24 completing — parser tests must be verified clean before auditing parser behavior
- Phase 26 (Companies) depends on Phase 24 — company backfill tests must be isolation-verified
- Phase 27 (Scoring) depends on Phase 24 — scoring tests must be clean before modifying scorer behavior

Phases 25 and 26 are independent of each other and could theoretically run in parallel, but sequential execution (25 -> 26) is simpler.

- Resume generation (Phase 4 of original roadmap) — separate milestone
- Intelligence features (Phase 5 of original roadmap) — separate milestone
- Scoring evaluation re-run (would be v1.4 to validate C1-C4 impact)
- New parser development (no new email sources)
- ATS scanner retry enhancement (DEBT-01, separate work)
