## Phase 15: Parser Fixes

**Goal**: Restore Glassdoor and Indeed email parsing, which currently extract 0 jobs due to format drift.

**Todos addressed**:
- Fix Glassdoor parser CSS class drift
- Fix Indeed parser email format change

### Plan 1: Glassdoor Parser

**Problem**: CSS class constants (`COMPANY_CLASS = "gd-628b46d9ce"`, `TITLE_CLASS = "gd-6c2846d4dc"`, `DETAIL_CLASS = "gd-28d35bae2f"`) have drifted. Parser's own drift detection confirms 0 extractions from 9+ emails.

**Approach**:
1. Inspect archived emails in `data/parse_failures/glassdoor_com_*.html` (8+ samples available)
2. Extract current CSS class names from the HTML
3. Update the three class constants in `glassdoor_parser.py`
4. Add structural fallback: if class-based extraction yields 0 jobs, try positional extraction:
   - Find `<a>` tags linking to `glassdoor.com/partner/jobListing`
   - Walk DOM tree (parent `<td>` or `<div>`) to find first span (company), first `<p>` (title), text with location/salary patterns
   - This makes the parser resilient to future class rotations

**Files**: `job_finder/parsers/glassdoor_parser.py`

### Plan 2: Indeed Parser

**Problem**: All Indeed alert emails return 0 jobs. The parser has multi-layer resilience (plain-text primary, HTML link fallback, HTML card fallback) but the email format has changed enough that all layers fail.

**Approach**:
1. Inspect archived emails in `data/parse_failures/indeed_com_*.html` (7+ samples available)
2. Compare against expected format (header line, `engage.indeed.com` delimiters, title/company/location blocks)
3. Update selectors and patterns to match the new format
4. If format diverged significantly, add a new strategy layer alongside existing ones (preserve backward compat)

**Files**: `job_finder/parsers/indeed_parser.py`

**Success criteria**: Parsers extract jobs from the archived failure emails. Existing tests still pass.

## Phase 16: Homepage Discovery

**Goal**: Unblock homepage resolution for 230+ miss companies and 71 hit companies with compound slugs.

**Todos addressed**:
- Add domain-guessing heuristic for company homepage discovery
- Fix compound slug homepage resolution
- Replace DDG with reliable search API for homepage discovery
- Separate homepage discovery into standalone scheduler job

### Plan 1: Heuristics (Domain-Guessing + Compound Slug Fix)

**Domain-guessing heuristic (Tier 1.5)** — new step in `discover_homepage()` between slug heuristic (Tier 1) and web search (Tier 2):

1. Normalize company name: lowercase, strip suffixes (Inc, Corp, LLC, Co, Ltd), remove spaces and special characters
2. `HEAD https://{normalized}.com` with `allow_redirects=True`, timeout 10s
3. Validate: 200 status, `text/html` content-type, not parked domain (reuse `_PARKED_DOMAIN_SIGNATURES`)
4. Return validated URL on success, fall through to next tier on failure

**Compound slug fix** — enhance `_try_slug_heuristic()` with normalization variants:

1. Strip hyphens: `hinge-health` → try `hingehealth.com`
2. Strip common suffixes: `frontcareers` → try `front.com` (strip `careers`, `jobs`, `hiring`, `team`)
3. Try each variant with same HEAD + parked-domain validation
4. Keep slug-as-is as first priority; variants are ordered fallbacks

**Files**: `job_finder/web/homepage_discoverer.py`

### Plan 2: Google Custom Search API Integration

Replace `_search_ddg()` with `_search_google_cse()`:

1. Add `google_cse` config section to `config.yaml` (API key + search engine ID)
2. New function `_search_google_cse(company_name) -> str | None`:
   - GET `https://www.googleapis.com/customsearch/v1` with `q="{company_name}" official website`
   - Parse response items, filter out Wikipedia/LinkedIn (existing `_SKIP_DOMAINS`)
   - Validate first candidate URL via HEAD request
3. Rate limiting: track daily query count in a `cse_daily_usage` row in the `app_state` table (date + count). Stop searches when 100/day free tier exhausted.
4. Fallback: if Google CSE fails (quota, network), log and return None (don't crash discovery)
5. Remove DDG code (`_search_ddg`, `_DDG_HTML_URL`, `_DDG_DELAY`)

**Config addition** (`config.yaml`):
```yaml
google_cse:
  api_key: "..."
  search_engine_id: "..."
```

**Files**: `job_finder/web/homepage_discoverer.py`, `config.yaml`, `config.example.yaml`

### Plan 3: Standalone Scheduler Job + Probe Tracking

1. Add `homepage_probe_attempted_at` column to `companies` table (new migration)
2. New scheduler job via `_make_tracked_job`: `homepage_discovery` running daily at 6:30 AM (before ATS scan at 7 AM)
3. Query: `companies WHERE homepage_url IS NULL AND (homepage_probe_attempted_at IS NULL OR homepage_probe_attempted_at < date('now', '-7 days'))`
4. Update `homepage_probe_attempted_at` after each probe attempt (success or failure)
5. Keep inline call in `run_ats_scan()` as secondary trigger
6. Batch cap remains at 50 per run

**Files**: `job_finder/web/homepage_discoverer.py`, `job_finder/web/scheduler.py`, `job_finder/web/db_migrate.py`

**Success criteria**: Domain-guessing catches simple `{company}.com` cases. Compound slug normalization resolves cases like `hinge-health` → `hingehealth.com`. Google CSE handles remaining misses within free tier. Probe tracking prevents redundant API calls. Standalone job runs independently of ATS scan.

## Phase 17: Code Quality

**Goal**: Fix misleading error attribution and stop test pollution of production logs.

**Todos addressed**:
- Separate template rendering from scan exception handler
- Isolate test logging from production log file

### Plan 1: Both Fixes

**Fix 1: Scan exception handler** (`companies.py:280-293`)

Split the single try/except that wraps both `run_ats_scan()` and `render_template()`:

```python
# Run scan in its own try/except
try:
    result = run_ats_scan(...)
    error = None
except Exception as e:
    result = None
    error = str(e)
    logger.exception("ATS scan failed")

# Render template outside scan try/except
# Flask handles template errors via its own error handler
return render_template("companies/_scan_result.html", result=result, error=error)
```

Add `| default(0)` to `result.html_scraped` in `_scan_result.html` for defensive rendering.

**Fix 2: Test log isolation** (`tests/conftest.py`)

After `create_app()` in the `app` fixture, strip the `RotatingFileHandler`:

```python
import logging
from logging.handlers import RotatingFileHandler
root = logging.getLogger()
root.handlers = [h for h in root.handlers if not isinstance(h, RotatingFileHandler)]
```

No changes to production code. Tests stop polluting `logs/app.log`.

**Files**: `job_finder/web/blueprints/companies.py`, `job_finder/web/templates/companies/_scan_result.html`, `tests/conftest.py`

**Success criteria**: Scan errors and template errors logged separately. Test runs produce no entries in `logs/app.log`. All existing tests pass.

## Phase 18: UI Improvements

**Goal**: Fix date filter UX and convert synchronous sync to async with progress feedback.

**Todos addressed**:
- Fix job board not refreshing when date filter is cleared
- Add async sync with progress feedback

### Plan 1: Date Filter Fix

**Problem**: Clearing a date input doesn't consistently fire `change` event across browsers, so the HTMX form doesn't re-fetch the job list.

**Approach**: Add explicit "Clear" buttons next to each date input:
1. Small "x" button adjacent to each `<input type="date">`
2. On click: set input value to `""`, fire `htmx.trigger(form, 'change')`
3. Also add `input` event to the HTMX trigger for belt-and-suspenders: `hx-trigger="change from:select, change from:input, input from:[type=date]"`

**Files**: `job_finder/web/templates/jobs/index.html`

### Plan 2: Async Sync with Progress Feedback

Convert synchronous `POST /dashboard/sync` to background + polling, following the batch-score pattern:

**Database**:
- New table `sync_sessions` (id, status, started_at, finished_at, phase, detail, jobs_fetched, new_jobs, errors)
- Migration added to `db_migrate.py`

**Routes**:
- `POST /dashboard/sync` → spawn `_sync_bg(db_path, session_id, config)` in background thread, return progress fragment immediately
- `GET /dashboard/sync/status/<session_id>` → return HTMX fragment with current progress

**Background function** (`_sync_bg`):
- Update `sync_sessions` at each phase transition:
  - "Fetching Gmail..." → "Parsing emails (N/M)..." → "Running SerpAPI..." → "Scoring new jobs..."
- Check `status='cancelling'` between phases for cancellation support
- On completion: mark session done with summary counts

**Progress fragment**:
- Shows current phase + counts
- Self-polls via `hx-trigger="every 2s"` until complete
- On completion: shows summary (jobs found, new, errors) with auto-dismiss after 10s

**Sidebar integration**:
- Replace `hx-swap="none"` on sidebar sync button
- Show small progress indicator inline when sync is running

**Files**: `job_finder/web/blueprints/dashboard.py`, `job_finder/web/templates/dashboard/index.html`, `job_finder/web/templates/components/_sidebar.html`, `job_finder/web/db_migrate.py`, new template fragment for sync progress

**Success criteria**: Date filter clear refreshes the job list. Sync runs in background with visible progress. User can navigate freely during sync. Sidebar sync button shows active state.
